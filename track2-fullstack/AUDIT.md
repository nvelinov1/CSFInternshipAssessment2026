# Audit - Application Issues Identified and Prioritized

This document contains a list of issues identified with the FarmTracker application. The issues are sorted by priority, which is determined by the severity if left unresolved and impact to users. 

### 1. ``/animals.html`` - Listing displays paddock ID instead of paddock name

**Description:** Navigating to the Animals listing incorrectly displays the paddock ID value instead of the paddock name. This information is not useful to display to the user.

**Priority**: Ranked 1st, as it prevents users from accurately identifying which animal belongs to which paddock.

### 2. ``/animals.html`` - Incorrect pagination on animals listing

**Description:** Clicking on the ``Next``/``Previous`` buttons to change the page number only results in a change of 1 animal record, rather than displaying the next page of 5 animals. The pagination should be updated so that 5 different animals are displayed when changing pages.

**Priority**: Ranked 2nd, as it prevents users from easily navigating through the listing to find an animal.

### 3. Database files committed to version control

**Description:** The ``.db`` files located in ``/app/backend`` have been committed to the Git repository. This is bad practice as these files represent artifacts generated by SQLite, which Git cannot track the changes for effectively. These files should be deleted from the repository, and a ``.gitignore`` file should be added to prevent developers from re-committing these files.

**Priority**: Ranked 3rd, as it prevents developers from easily managing files in source control.

### 4. ``/animals.html`` - Animals listing mobile display

**Description:** The animals listing overflows on mobile screen sizes (<500px width), causing the page to display in a disorganized manner. The listing is not mobile responsive on smaller screen sizes.

**Priority**: Ranked 4th, as it prevents mobile users from accessing the animals listing. However since it only relates to the UI, it can be prioritized lower. 

### 5. Ambiguous frontend date display

**Description:** The dates for heath events and dates of birth are displayed ambiguously. Users may interpret the date ``2026-01-02`` as January 2nd or February 1st. Changing this format to ``January 2nd, 2026`` results in better date display. 

**Priority**: Ranked 5th, as it leads to ambiguity with date representations.

### 6. ``POST /api/animals/`` - Status code

**Description:** The ``POST /api/animals/`` endpoint should be updated to return status code 201 (instead of 200) to indicate a created resource. This ensures consistency with ``POST /api/animals/:id/health-events``, which returns status code 201.

**Priority**: Prioritized last, as it is a minor change, and is unrelated to the application logic.
